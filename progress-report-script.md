# Note 项目进度汇报讲稿

本文是一份可直接用于口头汇报的中文讲稿，时长大约 30 分钟。内容按“项目背景、功能完成情况、技术实现、当前进度与后续计划”展开，尽量兼顾业务表达和技术表达。如果现场需要演示，可以按文中的“演示提示”来切换页面。

## 一、开场与项目定位

各位好，今天我汇报的项目是 `note`。这是一个基于 Go 和 Vue 实现的本地化智能笔记系统。我们希望解决的核心问题，不只是“把文字记下来”，而是让笔记具备三个能力：第一，信息可以被持续积累；第二，内容之间可以建立结构化连接；第三，在内容积累到一定规模之后，系统能够基于已有笔记做检索和问答。

从产品定位上看，这个项目参考了 Notion 一类工具的使用方式，但并不是简单做一个页面外观的模仿，而是围绕“本地可用、结构清晰、后续可扩展到智能检索”来搭建。也就是说，我们既做了笔记管理的基础能力，也把后续智能化的底层链路提前打通了。

这个项目目前的目标可以概括为一句话：先把一个可用、可演示、可继续演进的知识管理原型跑通。在这个阶段，我们特别关注三件事。第一，用户体验上是不是顺手，录入和查找信息是不是足够快。第二，技术架构上是不是清晰，前后端和存储是否便于继续扩展。第三，智能能力上是不是已经形成闭环，也就是笔记写进去之后，能不能被切块、向量化、检索并参与问答。

如果用一句更口语化的话来介绍，我会说：`note` 不是一个只会“存文档”的工具，而是一个把笔记、目录、标签、搜索、内部链接、收藏、语音输入和 RAG 问答放到同一个工作流里的项目。

## 二、项目当前已经完成的核心功能

下面我分功能层面介绍一下当前已经完成的内容。为了便于大家理解，我会从用户最直观能感受到的部分开始讲。

### 1. 笔记的创建、编辑、保存和删除

首先，最基本的笔记 CRUD，也就是创建、读取、更新和删除，已经是完整可用的。用户可以新建页面，编辑标题和 Markdown 内容，保存之后会落到本地 SQLite 数据库里。对于已经存在的页面，系统支持修改内容、重新保存，也支持删除。

这里面有一个体验上的重点，就是自动保存。当前系统在已有页面的编辑场景下，不要求用户每次都手动点击保存。前端会监听标题、正文、父子关系和标签这些关键字段的变化，然后通过一个带防抖的自动保存机制，在用户停止输入后自动把变更写回后端。这样做的目的，是尽量降低用户的心智负担，让记录过程更流畅。

当然，系统也保留了显式保存按钮，以及 `Ctrl/Cmd + S` 快捷键。也就是说，它同时照顾了“轻量记录”与“明确控制”两种习惯。

### 2. 阅读模式、Markdown 编辑与实时预览

第二项已经完成的能力，是阅读模式和编辑模式的分离。当前系统在打开一篇笔记时，默认进入的是阅读模式，页面直接展示 Markdown 渲染后的正式内容；当用户点击“编辑”之后，才会进入左右分栏的编辑和预览界面。这样做的好处，是把“阅读”和“写作”两个动作分开，避免用户一打开页面就看到编辑器，从而更接近真实笔记工具的使用习惯。

在编辑模式下，左侧输入 Markdown 内容，右侧可以看到解析后的 HTML 预览。这样既满足了技术用户对 Markdown 的偏好，也能让内容在录入过程中随时检查格式是否正确。

这个功能的重要性不只是“好看”，而在于它让笔记有了结构表达能力。比如标题、列表、代码块、引用、表格，这些内容都可以用统一的文本格式去承载。对于技术文档、学习笔记、工作记录来说，这比纯文本更有组织性，也更方便后续检索和再利用。

### 3. 页面树、文件夹和层级管理

第三项功能是层级化管理。当前系统支持父子页面关系，也支持把某个页面标记为文件夹，形成类似知识目录树的结构。用户可以在左侧看到一个树形的私有空间列表，上面可以展开、折叠，也可以在根目录下或者某个文件夹下新建页面。

这部分对知识管理非常关键。因为笔记一旦超过几十篇，如果只是平铺罗列，后续查找的成本会很高。通过父子结构和文件夹概念，系统把“信息堆积”变成了“信息分层”，也让用户可以按主题、项目、模块来组织自己的知识。

除了基础的层级展示之外，当前还支持通过拖拽来移动页面，把一个页面移动到另一个页面或者文件夹下面。这意味着结构调整是低成本的，用户不需要通过繁琐表单去修改层级，而是可以通过更直观的交互完成知识整理。

### 4. 标签体系

第四项功能是标签管理。用户可以给页面打标签，系统支持新增标签、选择已有标签、移除标签，也支持全局删除某个标签。标签的作用，是在目录结构之外提供另一种横向组织方式。

为什么说标签很重要？因为目录更像是“树状分类”，而标签更像是“多维索引”。比如一篇文章可能既属于“后端工程”目录，又同时带有“go”“database”“performance”三个标签。这样用户之后搜索某一类主题时，就不一定非得沿着目录一层层进入，可以从另外一个维度把内容聚合起来。

### 5. 搜索能力

第五项功能是搜索。当前系统支持搜索弹窗，用户可以通过 `Ctrl/Cmd + K` 快速打开搜索界面。在搜索结果中，系统支持按相关度排序，也支持按更新时间排序、按标题过滤、按正文过滤，并且还做了结果分组，比如今天、近 30 天、更早等分类。

这部分的价值是非常直接的。随着笔记增多，用户的核心诉求会逐渐从“写得进去”转向“找得出来”。搜索功能决定了系统能不能从一个记录工具，进化成一个真正能服务日常工作的知识库。

当前搜索在前端层面已经具备比较完整的交互体验，包括关键词高亮、搜索结果摘要、键盘定位等。后端也提供了查询接口，能够支撑按标题和内容做匹配。

### 6. 最近访问与收藏

第六项功能，是最近访问和收藏。首页有“最近访问”的卡片区域，会把用户近期打开过的笔记展示出来；同时也支持把页面加入收藏，便于高频访问内容快速打开。现在收藏能力不仅体现在首页收藏栏里，也在页面操作区提供了直接的“收藏/取消收藏”按钮，用户在阅读或编辑笔记时就可以即时管理常用页面。

这两个能力看起来不像“核心功能”，但它们对真实使用体验影响很大。因为在多数知识工具里，用户最常处理的并不是“所有内容”，而是一小部分当前阶段最相关的内容。最近访问和收藏，本质上是在为高频工作流做优化，让用户减少重复查找。

当前这部分数据通过浏览器本地存储来实现，属于轻量级但高性价比的做法。它不需要专门增加复杂的后端状态管理，却能明显提升用户的连续使用体验。

### 7. 复制、导出与撤回

第七项功能是页面复制、Markdown 导出，以及编辑过程中的撤回与重做。复制可以帮助用户基于某个已有页面快速派生新内容，比如周报模板、会议记录模板、技术方案模板。导出 Markdown 则保证了内容的可迁移性，避免信息完全被锁定在系统内部。

除了复制和导出之外，当前编辑器还补上了 `Ctrl+Z` 撤回、`Ctrl+Shift+Z` 和 `Ctrl+Y` 重做能力。这个能力不仅支持普通手动输入，也支持对语音输入和 AI 优化后产生的内容变更进行撤回。也就是说，系统在“写作辅助”增强之后，并没有牺牲编辑可控性。

### 8. 页面内引用与内部链接

第八项功能是内部链接能力。当前系统支持把页面之间建立跳转关系。一种方式是通过 `[[页面标题]]` 这样的 wiki link 风格语法；另一种方式是输入 `[显示名称](文件夹/文件夹/文件名)` 这样的路径式内部链接。系统会在预览和渲染时把它们解析成内部跳转链接，同时保留 Markdown 原始写法，避免用户在编辑区看到难理解的内部 ID。

这个功能对于知识系统特别重要，因为它把原本孤立的页面串成了网络。目录结构解决的是“上下级关系”，标签解决的是“横向分类”，内部链接解决的则是“语义关联”。一旦三者结合起来，系统中的笔记就不再只是文档列表，而开始具备知识网络的雏形。

### 9. AI 问答面板

第九项功能，是当前项目最有特色的一部分，也就是 AI 问答面板。页面右下角有一个 AI 入口，打开之后用户可以输入问题，系统会结合笔记内容给出回答。它不是单纯把用户问题直接发给大模型，而是先走一套本地笔记检索流程，再把检索到的上下文提供给模型，让模型在已有笔记范围内生成答案。

这也是我们把项目从“笔记工具”推进到“智能知识系统”的关键一步。因为一旦问答链路跑通，用户未来不只是自己去翻找历史笔记，也可以直接向系统提问，让系统帮助组织和提炼内容。

### 10. 语音输入与 AI 优化

第十项功能，是写作辅助能力。当前编辑模式已经支持语音输入，用户点击按钮后可以开始听写，并且支持暂停、继续和结束插入。这样一来，笔记记录不再局限于键盘输入，在会议记录、灵感记录或者快速整理思路时会更方便。

除了语音输入，系统还提供了“AI 优化”按钮。这个按钮会读取当前笔记内容，并结合笔记中已经链接到的关联页面内容，调用模型对 Markdown 进行重新排版和组织。这里的重点不是“让 AI 改写观点”，而是让 AI 帮助做结构整理，比如补标题层级、调整段落、提炼列表、让整篇笔记更易读。也就是说，它更像一个智能整理助手，而不是代替用户写内容。

### 11. 块级能力预留

除了用户已经能直接看到的这些功能，后端还预留了一套块级内容能力。也就是一篇笔记不仅可以整体以 Markdown 存储，还支持拆成 block，例如标题块、待办块、代码块、引用块、表格块等。当前后端已经有对应的数据表和 API，也具备 Markdown 与 block 之间的转换逻辑。

这部分虽然前端还没有完全切换成 block editor 形态，但从架构角度非常有价值。它意味着系统后续可以从“Markdown 编辑器”继续演进到“更细粒度的块编辑器”，例如支持块拖拽、块级引用、块级同步等更复杂的能力，而不需要推倒重来。

## 三、技术架构和实现方式

前面讲的是“做了什么”，下面我重点讲“这些功能是怎么实现的”。

### 1. 总体架构

整个项目采用的是比较清晰的前后端分层结构。前端用 Vue 3 和 Vite，后端用 Go，数据层使用 SQLite。本地页面由 Go 服务统一托管，前端打包之后输出到 `web/static`，后端除了提供 API，也负责静态文件服务。这样部署形式比较简单，适合当前原型阶段和本地化使用场景。

从分层来看，大致可以拆成四块：

第一块是前端交互层，负责页面展示、阅读与编辑切换、搜索弹窗、AI 面板、最近访问和收藏、语音输入以及 AI 优化等。

第二块是 HTTP API 层，负责把前端请求路由到具体业务逻辑，例如笔记的增删改查、标签管理、导出、Markdown 渲染、AI 优化以及 RAG 检索和问答。

第三块是存储层，也就是 SQLite 和 `store` 包，负责数据库 schema 初始化、兼容旧数据的迁移，以及各种实体的读写。

第四块是智能检索层，也就是 `rag` 和 `llm` 相关模块，负责切块、向量化、相似度计算、上下文拼装以及调用大模型生成回答。

### 2. 前端技术方案

前端使用的是 Vue 3 单文件组件方案，项目里核心界面主要集中在一个 `App.vue` 文件中，通过 `ref`、`computed`、`watch` 等响应式能力来组织状态。构建工具使用 Vite，这样开发启动快，打包也比较轻。

在交互实现上，前端做了不少细节工作。比如：

编辑器内容变化之后，会先做预览渲染，再在适当时机触发自动保存。

打开页面时默认进入阅读模式，点击编辑后才进入左右分栏的编辑器。

搜索框打开后会自动聚焦，键盘快捷键可以直接进入搜索流程。

最近访问和收藏通过浏览器 `localStorage` 做持久化，避免每次刷新都丢失。

树形页面列表支持展开折叠和拖拽移动。

上下文菜单支持重命名、移动、收藏、在新标签页打开等操作。

编辑模式支持语音输入、AI 优化，以及 `Ctrl+Z` / `Ctrl+Y` 这类撤回与重做操作。

这些实现并不是非常复杂的框架技巧，但它们共同构成了一个“可连续使用”的产品体验。换句话说，当前前端不是只把接口接上，而是已经尽量把用户常见操作路径走通了。

### 3. 后端技术方案

后端采用 Go 实现，入口在 `cmd/server`。服务启动时会先初始化数据库、加载模型提供者、构建 RAG 服务，然后注册 API 路由并启动 HTTP 服务。路由层使用的是 `chi`，它的特点是足够轻量，结构清晰，比较适合这种中小型服务。

Markdown 渲染使用的是 `goldmark`。这样可以把用户输入的 Markdown 转成 HTML，供前端预览和页面展示使用。

数据库选型是 SQLite，并且通过 `modernc.org/sqlite` 驱动来接入。这个选择和项目定位是一致的，因为当前项目强调本地化、轻部署和快速原型验证，SQLite 在这个阶段是很合适的。它不需要额外数据库服务，落地成本低，同时又足够支持笔记、标签、块、向量等数据结构。

在 schema 设计上，当前主要有几张表：

`notes` 用来存笔记基本信息，包括标题、Markdown、HTML、父页面、置顶状态、归档状态和时间戳。

`note_tags` 用来存笔记和标签的关系。

`note_chunks` 用来存 RAG 切块后的内容和对应向量。

`note_blocks` 用来支持块级内容表示。

另外，后端在启动时会做兼容型迁移检查，比如某些旧数据库文件里如果没有 `parent_id`、`is_pinned`、`is_archived` 或 `note_blocks.level` 这些字段，系统会通过 `ALTER TABLE` 自动补齐。这意味着项目并不是一次性写死 schema，而是已经考虑到了后续演进和旧数据兼容。

### 4. API 设计

在接口层，当前后端已经提供了比较完整的 REST 风格 API。包括：

笔记列表、单笔记详情、创建、更新、删除；

复制、导出；

标签列表和标签删除；

Markdown 渲染与内部链接解析；

块级内容读取和替换；

AI 优化接口；

RAG 搜索和 RAG 问答。

这样的接口设计使得前端和后端职责划分比较清楚。前端更多负责状态管理和交互编排，后端负责数据一致性、渲染、索引和模型调用。后续如果要接新的前端，甚至做桌面端或移动端，也有较好的复用空间。

## 四、智能能力是如何落地的

接下来我单独讲一下智能检索和问答部分，因为这是这个项目技术含量相对更高、也更适合在汇报里体现亮点的部分。

当前项目采用的是典型的 RAG，也就是检索增强生成思路。整体流程可以分成四步。

第一步，用户写入笔记。笔记保存成功之后，后端会把 Markdown 内容拿出来。

第二步，对内容进行切块。也就是把长文本按一定长度拆成若干 chunk。当前配置里有最大字符数控制，目的是让每个片段既保留足够语义，又不要太长。

第三步，对每个 chunk 做向量化。系统会调用 embedding 模型，把文本转成向量，然后把向量和对应内容一起存入 `note_chunks` 表。

第四步，在问答阶段，先把用户问题也转成向量，再和所有 chunk 做相似度计算，选出最相关的 top-k 片段。然后系统把这些片段组织成 prompt，发给生成模型，最后得到答案。

在这里，我们没有直接依赖某一个固定大模型，而是做了提供方抽象。当前支持两种模型来源。

一种是 OpenAI 兼容接口模式，默认可接 DashScope 这类兼容 OpenAI API 的服务。这个模式同时支持 embedding 和 chat completion。

另一种是 Ollama 模式，适合本地模型部署。这样做的价值在于，项目既可以连接云端大模型，也可以在本地环境下运行，具备比较好的灵活性。

从工程实现上看，`llm` 模块分别封装了 OpenAI 兼容客户端和 Ollama 客户端；`rag` 模块则封装了索引、搜索和问答流程。两者之间通过接口耦合，而不是写死到某一个 SDK。这样做的好处，是后续如果要切换模型、增加重排序、混合检索，改动范围会比较可控。

另外，项目里还实现了一个 `seednotes` 工具，用于批量导入演示数据。这个工具会创建多个目录和较长的示例笔记，并且在配置好模型后自动重建索引。对于项目演示、测试和答辩来说，这一点很实用，因为它解决了“系统是空的，不好演示”的问题。

## 五、当前项目的完成度和价值

如果从“能不能跑起来”这个层面看，这个项目已经远远超过了一个静态 Demo。它现在具备的是一套真实可操作的闭环：

内容可以被创建和组织；

内容可以被搜索和跳转；

内容之间可以建立内部引用；

内容可以被复制、导出，并通过收藏、最近访问和内部链接快速复用；

内容还可以进一步被切块、向量化，并用于问答。

也就是说，从“记录”到“管理”再到“智能利用”，这三层能力已经基本衔接起来了。对于一个阶段性项目来说，我认为这是当前最大的进展。

如果从可扩展性来看，项目也已经预留了几条很明确的后续路径。

第一条路径，是编辑器能力继续增强。因为后端已经有 block API，后续可以往块编辑器方向升级。

第二条路径，是搜索能力继续增强。当前已经有普通搜索和向量检索，未来可以进一步做 FTS5、更细的排序优化，甚至接入更强的检索引擎。

第三条路径，是 AI 能力继续增强。现在已经不只是问答，还加入了笔记排版优化。未来还可以扩展到自动总结、标签建议、知识图谱式关联推荐等。

第四条路径，是多端与部署形态扩展。当前是本地 Web 形态，后续可以考虑桌面封装、容器化、远程同步等。

## 六、目前也存在的边界和下一步计划

最后我也想客观说明一下，当前项目虽然已经形成了完整雏形，但仍然处在原型到增强版本之间的阶段，还有一些边界是明确存在的。

第一，前端目前仍然是以单页和集中状态管理为主，优点是开发效率高、迭代快，缺点是随着功能继续增加，后续可能需要进一步拆分组件，提升可维护性。

第二，当前块级编辑能力主要还是后端预留，前端主体编辑体验仍然是 Markdown 双栏模式。这对技术用户很友好，但如果面向更广泛用户群体，后续还可以继续优化编辑器交互。

第三，RAG 问答已经跑通，但评价体系、命中率分析、答案质量标注、引用来源展示等方面还可以继续加强。也就是说，我们已经有“能工作”的智能能力，下一步要做的是把它打磨成“更稳定、更可解释”的能力。

第四，目前是本地 SQLite 存储，适合单机原型和轻量使用场景。如果未来要走多用户、协作、云端部署路线，就需要进一步考虑鉴权、并发写入、同步与权限控制。

基于这些现状，我认为下一步比较务实的计划可以是三件事。第一，继续优化编辑和搜索体验，把常用工作流再打磨顺。第二，补充更多 RAG 质量验证和可观测能力，让智能部分不仅能演示，也更可信。第三，把 block editor 或更细粒度内容组织能力逐步前移到前端，增强系统的结构化表达能力。

## 七、结尾总结

最后我做一个总结。

`note` 这个项目当前已经完成了一个比较完整的本地智能笔记系统雏形。它不是只做了页面展示，而是把笔记录入、阅读与编辑切换、结构化管理、标签、搜索、内部链接、收藏、语音输入、AI 优化和智能问答这些能力串成了一个闭环。技术上，项目使用 Go、Vue 3、Vite、SQLite、Chi、Goldmark 以及可切换的大模型接入方案，整体架构清晰，后续扩展空间也比较明确。

如果用一句话概括当前进度，我会说：这个项目已经从“想法验证”走到了“功能闭环验证”，下一阶段的重点不再是有没有，而是做得够不够稳定、够不够顺手、够不够智能。
